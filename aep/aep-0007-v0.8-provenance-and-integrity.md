# AEP-0007: v0.8 Provenance and Integrity Artifacts - Evidence for E2E Chains

- Status: Draft
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.8.0
- Related: AEP-0001 (Accepted), AEP-0006 (Accepted), CONFORMANCE Level 5

## 1. Abstract

v0.8.0 upgrades AOP's end-to-end artifact chain
(`resolve -> fetch -> evaluate -> decision`) by adding verifiable
provenance and integrity evidence artifacts:

- provenance fields (who, what, when, how)
- integrity references (hash and signature pointers)
- chain binding (evidence references exact manifest/policy/decision)

This AEP remains runtime-agnostic. It defines schema-backed attestation
artifacts, fixtures, and CI gates without introducing a reference
runtime.

## 2. Motivation

AOP v0.7.0 proves semantic consistency across artifacts via positive and
negative E2E gates. The next barrier is trust:

- How can consumers verify where an artifact came from?
- How can consumers verify that evidence binds to the exact chain
  artifacts used for decisions?

Supply-chain standards provide a mature reference model:

- in-toto uses layout and link metadata to provide a verifiable
  evidence chain.
- SLSA provenance defines structured metadata for how artifacts are
  produced.

v0.8 captures these ideas as schema-backed AOP evidence artifacts.

## 3. Scope and non-goals

### In scope (v0.8.0)

- New schema family for AOP evidence and attestation artifacts
  (normative)
- E2E fixtures extended with evidence binding fields
- CI gates validating:
  - evidence schema validity
  - evidence binding to manifest/policy/decision IDs and digests
  - rejection of invalid evidence fixtures

### Non-goals

- Mandatory signature verification (hold for v1)
- Full key management and transparency log system (hold for v1)
- Network transport guarantees for artifact fetch

## 4. Proposed schema family

Add one new normative schema:

- `schemas/aop-evidence.schema.json`

AOP Evidence is an attestation envelope (signature optional in v0.8)
containing:

- `subject`: artifact reference (id + digest)
- `predicateType`: evidence type (e.g., provenance)
- `predicate`: structured provenance payload (minimal subset inspired by
  SLSA concepts)

Design constraints:

- Keep schema minimal and self-contained
- Reuse unknown-field discipline (`unevaluatedProperties`)

## 5. v0.8 fixture layout

- `examples/v0.8/e2e/positive/<scenario>/`
  - `resolve.json`
  - `manifest.json`
  - `policy.json`
  - `decision*.json`
  - `evidence.manifest.json`
  - `evidence.decision.json`
- `examples/v0.8/e2e/negative/<scenario>/`
  - same structure; each scenario violates exactly one
    evidence-binding rule

## 6. Cross-artifact binding rules (CI-enforced)

For each positive scenario:

1. Evidence subjects MUST bind to chain artifacts:
   - `evidence.manifest.subject.id == manifest.id`
   - `evidence.manifest.subject.digest.sha256` matches manifest digest
     (canonical JSON digest or precomputed fixture digest)
   - `evidence.decision.subject.id` matches a stable decision reference
     (`decision_id` or equivalent)
2. Evidence MUST reference the policy used:
   - `evidence.decision.predicate.policy_id == policy.id`
3. Evidence MUST NOT embed secrets.

Negative scenarios MUST include at least:

- one with wrong manifest digest
- one with wrong policy ID reference

## 7. Conformance integration

Add **Level 6 - Evidence-Bound E2E Chain** in `CONFORMANCE.md`.

Level 6 requires Level 5 plus:

- evidence artifacts validate against `schemas/aop-evidence.schema.json`
- evidence binding rules pass for positives
- evidence negative fixtures are rejected

## 8. CI plan

Add a v0.8 evidence gate that:

- validates all v0.8 files by schema family
- validates binding constraints across resolve/manifest/policy/decision
  and evidence artifacts
- rejects negative evidence scenarios

## 9. Acceptance criteria

AEP-0007 is accepted when:

- at least one positive v0.8 scenario includes manifest and decision
  evidence
- at least two negative evidence scenarios exist
- CI enforces evidence schema and binding rules
  (positive pass, negative reject)
- `CONFORMANCE.md` includes Level 6 with reproducible fixture anchors
