# AEP-0003: v0.4.0 Agenda - Policy Semantics, Registry Surface (Non-Normative), and MCP Interop Hardening

- Status: Draft
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.4.0
- Related: AEP-0001 (Accepted), AEP-0002 (Accepted), `CONFORMANCE.md`, SPEC bridge note

---

## 1. Abstract

This AEP proposes the v0.4.0 agenda for AOP after v0.3.x governance tightening and extension namespace enforcement.

v0.4.0 focuses on:

- Policy semantics that can be mechanically validated and gated by runtimes.
- A minimal registry surface as non-normative artifacts (docs + schemas) to enable discovery workflows without v1-level lock-in.
- MCP interop hardening around schema-backed structured tool output.

This AEP also defines a fixtures plan so each item can be tested through CI.

---

## 2. Normative Language

RFC 2119 keywords apply when referenced from SPEC normative sections. In this AEP, requirements are planned targets and become normative only when merged into SPEC and schema artifacts.

---

## 3. Motivation

v0.3.x established a strict schema-backed conformance baseline:

- Governance core fields are required when sections are present.
- Extension namespace is standardized.
- Unknown fields are rejected via `unevaluatedProperties`.
- Positive and negative fixtures are enforced in CI.

The next adoption barrier is runtime gating:

- Without policy semantics, runtimes cannot consistently enforce security and capability constraints.
- Without a discovery and registry surface, object distribution and resolution workflows remain ad hoc.
- MCP interop benefits from AOP schema discipline: MCP tools can publish output schemas, and consumers can validate responses against those contracts.

---

## 4. Scope and Non-Goals

### In Scope (v0.4.0)

- Policy object schema and minimal enforcement semantics hooks (machine-actionable gating).
- Registry surface documentation plus JSON Schemas for registry records and resolution responses (non-normative).
- MCP interop guidance and mapping examples (non-normative), with optional schema artifacts for validation helpers.

### Non-Goals (Deferred)

- Mandatory signature verification as a universal requirement (hold for v1).
- Full registry protocol guarantees (auth, transparency logs, immutability contracts) (hold for v1).
- Multi-file schema module stability guarantees across `$id` and `$anchor` boundaries (hold for v1).

---

## 5. Proposal A - Policy Semantics (Normative)

### 5.1 Policy object intent

Define `kind: "policy"` objects that express gating rules, at minimum:

- Required auth schemes (for example, oauth2 only).
- Required permissions shape and constraints.
- Capability thresholds (for example, `side_effects <= low`).
- Execution allow or deny based on invocation mode or determinism.

### 5.2 Policy evaluation contract (minimal)

A conforming runtime that supports policy enforcement:

- SHOULD provide a policy evaluation point before invocation.
- SHOULD treat policy outcomes as enforceable (allow or deny with reasons).
- MUST NOT treat policy as advisory-only when operating in policy-enforced mode.

### 5.3 Schema artifacts

Add `schemas/aop-policy.schema.json` and at least:

- `examples/aop-policy.allow.json`
- `examples/aop-policy.deny.json`

Add negative fixtures that prove invalid policy objects are rejected.

---

## 6. Proposal B - Registry Surface (Non-Normative for v0.4.0)

### 6.1 Registry record shape (schema-backed, non-normative)

Define a registry record schema that includes:

- object id
- object version
- digest or checksum metadata
- retrieval locations
- optional signature reference
- indexing metadata

### 6.2 Resolution response shape (schema-backed, non-normative)

Define a minimal resolve response schema that includes:

- requested id
- resolved manifest (or pointer)
- verification hints (`checksum`, `signature_ref`)
- typed errors for not found, invalid, or unauthorized

### 6.3 Why non-normative now

This enables interoperability experiments without prematurely locking v1-level stability guarantees.

---

## 7. Proposal C - MCP Interop Hardening (Non-Normative)

Provide a short interop guide and examples:

- Mapping AOP `tool` objects to MCP tools.
- Reusing AOP `schema.inputs` and `schema.outputs` as MCP tool schemas where applicable.
- Validation expectations aligned with MCP guidance when output schemas are provided.

---

## 8. Compatibility Impact (Expected v0.4.0 stance)

v0.4.0 is a MINOR release under v0.x. It MAY include breaking changes but MUST include:

- Explicit breaking-change disclosure.
- Migration guidance.
- Updated fixtures (valid + invalid) and CI gates.

Preferred direction: keep v0.4.0 additive where possible and avoid new required core-manifest fields unless strongly justified.

---

## 10. Alternatives Considered

- Skipping policy semantics: rejected because governance fields cannot be enforced consistently.
- Making registry normative in v0.4.0: rejected to avoid premature v1-level commitments.
- Treating MCP interop as normative in v0.x: rejected; guidance stays flexible during experimentation.

---

## 11. Security Considerations

- Policy objects reduce confused-deputy risk by making gating machine-actionable.
- Registry artifacts must avoid embedded secrets and use references plus runtime-secured configuration.
- Signature verification remains optional in v0.x, but data fields for verification experiments should be present.

---

## 12. Implementation Plan

- Draft policy schema plus positive and negative fixtures.
- Add non-normative registry schemas plus examples.
- Add MCP interop guide text and minimal mapping examples.
- Keep CI conformance strict (compile, validate, negative gate).
- Tag and release v0.4.0 with migration notes for any breaking changes.
