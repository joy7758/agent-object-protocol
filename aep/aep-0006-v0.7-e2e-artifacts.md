# AEP-0006: v0.7 End-to-End Artifacts - Resolve -> Fetch -> Evaluate -> Decision

- Status: Draft
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.7.0
- Related: AEP-0001 (Accepted), AEP-0003 (Accepted), AEP-0004 (Accepted),
  AEP-0005 (Accepted)
- Scope: Schemas + fixtures + CI semantic gates (no reference runtime)

---

## 1. Abstract

This AEP proposes v0.7.0 as an end-to-end interoperability milestone:
`resolve -> fetch -> evaluate -> decision`, expressed entirely through
verifiable artifacts.

v0.7.0 does not introduce a runtime implementation. It defines:

- E2E fixture sets (positive + negative)
- schema validation for each step
- semantic consistency gates across artifacts

---

## 2. Motivation

AOP already provides schema-backed contracts for:

- object manifests
- policy objects
- policy decision envelopes
- registry record / resolve response (non-normative but schema-validated)

What is still missing is a single reproducible chain that demonstrates
how these artifacts compose in practice. v0.7.0 addresses that with CI
verifiable end-to-end fixture scenarios.

This mirrors common policy-system decomposition:

- PEP (enforcement) requests decisions from PDP (decision) and enforces
  them.
- Policy decisions may return structured data beyond allow/deny.

---

## 3. Scope and non-goals

### In scope

- E2E fixtures for a minimal chain:
  1. resolve response (registry)
  2. fetched object manifest (object)
  3. policy input (policy)
  4. decision output (policy-decision)
- cross-artifact semantic constraints
- CI gate enforcing:
  - schemas compile
  - positive scenarios pass
  - negative scenarios are rejected

### Non-goals

- network transport protocol implementation details (HTTP auth, caching,
  mirrors)
- mandatory signature verification (hold for v1)
- full registry guarantees (immutability, transparency)

---

## 4. Proposed artifacts (v0.7)

### 4.1 Fixture directory layout

- `examples/v0.7/e2e/positive/<scenario>/`
- `examples/v0.7/e2e/negative/<scenario>/`

Each scenario contains:

- `resolve.json` (`aop-resolve-response`)
- `manifest.json` (`aop-object`)
- `policy.json` (`aop-policy`)
- `decision.<strategy>.json` (`aop-policy-decision`)

### 4.2 Cross-artifact semantic constraints (CI-enforced)

For each positive scenario:

1. `resolve.request.object_id` MUST equal `manifest.id`.
2. `decision.policy_id` MUST equal `policy.id`.
3. `decision.target.selector=object_id` and
   `decision.target.object_id` MUST equal `manifest.id`.
4. If `decision.effect=deny` and `decision.patches` exists, patch
   operations MUST be schema-valid and decision reasons SHOULD reference
   deny semantics.

For negative scenarios:

- at least one constraint is intentionally violated and MUST be
  rejected by semantic gate logic.

---

## 5. Conformance integration

- Introduce Level 5 in `CONFORMANCE.md`:
  - End-to-End Artifact Chain
- Level 5 requires:
  - all step schemas validate
  - semantic gate passes for positive scenarios
  - semantic gate rejects negative scenarios

---

## 6. CI plan

Add a dedicated E2E gate step:

- validate `resolve.json` against
  `schemas/aop-resolve-response.schema.json`
- validate `manifest.json` against `schemas/aop-object.schema.json`
- validate `policy.json` against `schemas/aop-policy.schema.json`
- validate `decision*.json` against
  `schemas/aop-policy-decision.schema.json`
- run cross-artifact semantic consistency checks
- enforce negative scenarios are rejected

This gate should follow existing repository practice:
schema-compile + positive-pass + negative-reject.

---

## 7. Security considerations

- E2E fixtures MUST NOT embed secrets.
- Unknown-field control discipline MUST remain strict
  (`unevaluatedProperties` and related schema controls).
- Future work may bind integrity/signature verification into the chain
  (v1 scope).

---

## 8. Acceptance criteria

AEP-0006 is ready for acceptance when:

1. at least one positive E2E scenario is added
2. at least two negative E2E scenarios are added
3. CI enforces schema + semantic gates for E2E scenarios
4. `CONFORMANCE.md` includes Level 5 with reproducible fixture anchors
