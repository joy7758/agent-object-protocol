# AEP-0008: v0.9 "v1 Pre-Freeze" Sprint

- Status: Accepted
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.9.0
- Related: AEP-0001 (Accepted), AEP-0006 (Accepted), AEP-0007 (Accepted)
- Scope: Schemas + fixtures + CI semantic gates (no reference runtime)

---

## 1. Abstract

v0.9.0 is a pre-freeze sprint to prepare AOP for v1.0.

It delivers three outcomes:

1. Evidence profile explicitness: define minimal, schema-validated
   evidence predicate profiles. v0.8 introduced evidence binding; v0.9
   defines what evidence means.
2. Registry -> E2E -> evidence convergence: unify resolve artifacts
   with E2E chains and evidence binding so the chain is self-contained
   and verifiable.
3. v1 public API freeze candidate: publish an explicit list of what
   AOP will treat as stable public API at v1.0, and what remains
   experimental, extension-only, or non-normative.

SemVer reminder: during `0.y.z`, anything may change; `1.0.0` defines
the public API.

---

## 2. Motivation

AOP v0.7.0 provided an end-to-end chain
(`resolve -> fetch -> evaluate -> decision`) with positive and negative
gates.

AOP v0.8.0 added evidence artifacts and enforced digest binding using
`SHA-256(JCS(json))` (RFC 8785), making integrity verifiable and
CI-adjudicable.

The remaining adoption blockers are meaning and stability:

- Evidence is bound, but not yet standardized into explicit profiles.
- The chain exists, but registry resolve and evidence are not yet fully
  converged into packaged scenario suites.
- External adopters still need a clear statement of what fields and
  behaviors are expected to be stable at v1.0.

v0.9.0 makes this the final stepping stone into a v1.0 freeze.

---

## 3. Scope and non-goals

### In scope (v0.9.0)

- Minimal evidence profiles (schema-backed):
  - `aop.provenance/v0` (AOP-native minimal provenance profile)
  - `slsa.provenance/v1` (compatibility mapping subset, not a verbatim
    copy)
- Converged E2E scenarios that include:
  - `resolve.json` (registry resolve response)
  - `manifest.json` (object)
  - `policy.json` (policy)
  - `decision*.json` (decision envelope)
  - `evidence.*.json` (evidence envelopes)
  - CI gates for schema, cross-artifact bindings, digest bindings, and
    profile checks
- v1 pre-freeze documentation with an explicit public API candidate
  list.

### Non-goals

- Mandatory signature verification and key management (hold for v1).
- Full registry protocol guarantees (immutability, transparency logs,
  authorization model).
- Runtime implementation.

## 4. Proposal A - Evidence Profile Explicitness (normative)

### 4.1 Add profile schemas (normative)

Add a new schema family:

- `schemas/profiles/aop-provenance.profile.schema.json`
- `schemas/profiles/slsa-provenance-min.profile.schema.json`

These schemas constrain `aop-evidence.schema.json` by:

- requiring `predicateType` to be a recognized profile identifier
- constraining `predicate` structure per profile
- preserving unknown-field discipline (`unevaluatedProperties`)

### 4.2 Profile identifiers (recommended)

- `predicateType = "https://aop.dev/profiles/provenance/v0"`
- `predicateType = "https://slsa.dev/provenance/v1"` (compat subset)

v0.9 MUST provide fixtures for at least one profile, and SHOULD provide
both.

### 4.3 Profile conformance rules (CI-enforced)

For each evidence artifact in v0.9 scenarios:

- `predicateType` MUST be recognized.
- `predicate` MUST validate against the profile schema.
- Subject bindings and digest bindings MUST still pass (v0.8 rules).

---

## 5. Proposal B - Registry -> E2E -> Evidence Convergence

### 5.1 Scenario packaging

Introduce v0.9 scenarios under:

- `examples/v0.9/e2e/positive/<scenario>/`
- `examples/v0.9/e2e/negative/<scenario>/`

Each scenario MUST include:

- `resolve.json`
- `manifest.json`
- `policy.json`
- `decision*.json`
- `evidence.manifest.json`
- `evidence.decision.json`

### 5.2 Cross-artifact binding rules (CI-enforced)

Positive scenarios MUST satisfy:

1. Resolve/manifest identity:
   - `resolve.request.object_id == manifest.id`
2. Decision/policy identity:
   - `decision.policy_id == policy.id`
3. Decision target binding:
   - `decision.target.selector == "object_id"`
   - `decision.target.object_id == manifest.id`
4. Evidence subject bindings:
   - `evidence.manifest.subject.id == manifest.id`
   - `evidence.decision.subject.id == decision.decision_id` (or stable
     decision reference)
5. Digest bindings (RFC 8785 JCS):
   - `evidence.manifest.subject.digest.sha256 ==
     SHA-256(JCS(manifest.json))`
   - `evidence.decision.subject.digest.sha256 ==
     SHA-256(JCS(decision*.json))`

Negative scenarios MUST each violate exactly one rule and MUST be
rejected by the semantic gate.

## 6. Proposal C - v1 Public API Freeze Candidate

### 6.1 Public API candidate list

Add a new document:

- `V1_PUBLIC_API_CANDIDATE.md`

It MUST enumerate:

- Stable candidates intended for v1 freeze:
  - object schema core (`aop-object.schema.json`) and governance fields
  - policy schema (`aop-policy.schema.json`)
  - decision envelope schema (`aop-policy-decision.schema.json`)
  - evidence envelope schema (`aop-evidence.schema.json`) and
    supported profile identifiers
  - conformance levels and their gate definitions (Levels 2-6, then
    Level 7 once merged)
- Explicit non-stable items:
  - registry schemas (remain non-normative until v1+ decision)
  - experimental profiles and `x-*` extension fields

### 6.2 Freeze discipline

v0.9 SHOULD add a lightweight CI check that fails if the public API
candidate list changes without an AEP reference.

## 7. Conformance integration

Add **Level 7 - Profiled Evidence-Bound E2E Chain** in
`CONFORMANCE.md`.

Level 7 requires Level 6 plus profile validation
(`predicateType` and `predicate`) in v0.9 scenarios. Negative profile
violations MUST be rejected.

## 8. CI plan (incremental gates)

v0.9 introduces:

- compile and validate gates for profile schemas
- v0.9 E2E gate with positive-pass and negative-reject behavior:
  - schema validation for all scenario files
  - semantic bindings (resolve, policy, target)
  - digest bindings (JCS + SHA-256)
  - profile checks

---

## 9. Fixtures plan (minimum)

### 9.1 v0.9 positive

- At least one positive scenario containing:
  - `resolve/manifest/policy/decision`
  - `evidence.manifest` and `evidence.decision`
  - evidence predicates that validate against a profile schema

### 9.2 v0.9 negative (minimum three)

- wrong resolve -> manifest identity mismatch
- wrong evidence digest (JCS hash mismatch)
- wrong evidence profile
  (`predicateType` recognized but predicate shape invalid)

---

## 10. Security considerations

- Evidence artifacts MUST NOT embed secrets.
- Unknown-field control MUST remain strict
  (`unevaluatedProperties` discipline).
- Signature packaging alignment MAY reference DSSE / in-toto envelope
  as a future direction, but signature verification remains out of
  scope for v0.9.
- Digest + canonicalization reduces ambiguity and supports tamper
  detection.
