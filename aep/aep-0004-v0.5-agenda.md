# AEP-0004: v0.5.0 Agenda - Policy/Decision Combination Semantics and End-to-End Interop

- Status: Draft
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.5.0
- Related: AEP-0001 (Accepted), AEP-0002 (Accepted), AEP-0003 (Accepted), `SPEC.md`, `CONFORMANCE.md`

---

## 1. Abstract

This AEP defines the v0.5.0 agenda for advancing AOP from schema-valid components to reproducible cross-runtime decision behavior.

v0.5.0 focuses on:

- Policy and decision combination semantics that are testable and comparable across implementations.
- Minimal end-to-end interoperability flow (resolve -> fetch -> evaluate -> emit decision), kept non-normative.
- MCP interop hardening through explicit schema-conformance references for structured outputs.

---

## 2. Normative language

RFC 2119 keywords apply only when merged into normative SPEC sections and schema artifacts. In this AEP, requirements are planning targets.

---

## 3. Motivation

v0.4.x introduced:

- Policy schema family (`aop-policy`)
- Policy decision envelope schema family (`aop-policy-decision`)
- Registry interoperability artifacts (`aop-registry-record`, `aop-resolve-response`)
- CI gates with positive and negative fixtures per schema family

What remains underspecified is cross-runtime behavioral comparability: two conforming implementations may emit different decisions for the same inputs if strategy semantics are not made explicit.

v0.5.0 addresses this by making strategy semantics and end-to-end evaluation flows testable without requiring a reference runtime implementation in this repository.

---

## 4. Scope and non-goals

### In scope (v0.5.0)

- Clarify and constrain rule-combination semantics for policy evaluation output.
- Strengthen decision-envelope expectations (for example, strategy traceability).
- Add end-to-end interoperability fixtures across registry, object, policy, and decision artifacts.
- Provide MCP mapping guidance for output schema validation responsibilities.

### Non-goals (deferred)

- Mandatory global policy engine behavior lockstep across all runtimes.
- Registry protocol guarantees at v1-level (immutability, transparency logs, trust roots).
- Cryptographic signing and verification mandates as default runtime requirements.

---

## 5. Proposal A - Policy/Decision Combination Semantics (Normative Target)

### 5.1 Strategy vocabulary hardening

Use the existing decision strategy vocabulary in `aop-policy-decision`:

- `deny_overrides`
- `allow_overrides`
- `first_match`
- `all_match`

These strategies are selected to align with common policy-combining
patterns and to keep cross-runtime comparison explicit in decision
outputs.

Target for SPEC in v0.5.0:

- Define minimum behavioral expectations for each strategy.
- Define tie/conflict behavior where multiple rules match.
- Define how empty-match sets are resolved with policy `default_effect`.

### 5.2 Decision traceability

Target for SPEC/schema alignment:

- Decision output SHOULD include `decision.strategy` when policy evaluation is performed.
- Decision output SHOULD include `decision.rule_ids` for matched or contributing rules.
- `reasons[]` should remain mandatory for auditability.

### 5.3 Conformance fixtures

Add paired fixtures that prove strategy differences are explicit and deterministic:

- same policy + object, different strategy -> expected decision deltas
- same strategy + reordered rules -> expected stable or documented behavior

---

## 6. Proposal B - End-to-End Interop Flow (Non-Normative)

Define a minimal workflow and fixtures:

1. Resolve object metadata (`aop-resolve-response`)
2. Fetch object manifest (object/policy)
3. Evaluate policy for a target object
4. Emit policy decision envelope (`aop-policy-decision`)

The flow remains non-normative in v0.5.0, but schema-validated examples and invalid fixtures MUST be provided for reproducibility and interoperability tests.

---

## 7. Proposal C - MCP Interop Hardening (Non-Normative)

Document explicit mapping guidance:

- AOP `schema.outputs` <-> MCP output schema where applicable
- Provider conformance and consumer validation expectations for structured outputs
- Decision-envelope usage as auditable runtime metadata alongside tool responses

Add examples that show an AOP tool and policy decision output represented in MCP-compatible structured payloads.

---

## 8. Compatibility impact (expected v0.5.0 stance)

v0.5.0 is a MINOR release under v0.x and MAY include breaking changes, but MUST include:

- explicit breaking-change notes
- migration guidance
- fixture updates for new semantics
- CI gates that demonstrate new constraints

Preferred direction: additive or narrowing changes with migration guidance; avoid unrelated core-manifest breaks.

---

## 9. Conformance and fixture plan (v0.5.0)

### 9.1 Schema changes

- Update `schemas/aop-policy-decision.schema.json` only if needed for strategy semantics (for example, stronger conditional requirements around `strategy` and `rule_ids`).
- Keep extension namespace and unknown-field rejection discipline across all families.

### 9.2 Positive fixtures

- `examples/v0.5/combo/object/*.json` and
  `examples/v0.5/combo/policy/*.json` for strategy inputs.
- `examples/v0.5/combo/expected/*.json` for expected decision envelopes.
- `examples/v0.5/mcp/tool/*.json` and
  `examples/v0.5/mcp/expected/*.json` for schema-derived MCP interop
  output validation.

### 9.3 Negative fixtures

- invalid strategy value or missing required strategy trace fields
- contradictory decision payloads (for example, deny with empty reasons)
- malformed interop linkage artifacts
- invalid MCP expected outputs under `examples/invalid/v0.5/mcp/*.json`

### 9.4 CI updates

- compile all schemas
- validate each family of positive examples by target schema
- reject all invalid fixtures by target schema
- keep strict family segmentation to avoid false positives

### 9.5 v0.5 initial fixtures plan (phase 1)

Directory skeleton:

- `examples/v0.5/combo/{object,policy,expected}`
- `examples/v0.5/mcp/{tool,expected}`
- `examples/invalid/v0.5/{combo,mcp}`

Phase-1 targets:

1. Combination semantics:
   - Provide one policy plus one object that yields different decisions
     under two strategies.
   - Provide two expected decision envelopes documenting the difference.
2. MCP interop:
   - Provide one AOP tool object and its mapped MCP tool definition.
   - Use a schema-derived output validation route:
     `tool.schema.outputs` is extracted and used to validate expected
     output payloads.
   - Provide one expected structured result shape aligned with output
     schema guidance, plus invalid outputs that must be rejected.

---

## 10. Alternatives considered

- Keep strategy semantics implementation-defined only:
  - rejected due to low cross-runtime comparability.
- Make end-to-end flow normative immediately:
  - rejected to avoid premature protocol lock-in.
- Delay MCP interop guidance:
  - rejected because schema-backed interoperability is already a core AOP value.

---

## 11. Security considerations

- Explicit strategy semantics reduce policy ambiguity and inconsistent enforcement risk.
- Decision traceability improves auditability and incident analysis.
- End-to-end fixtures help surface confused-deputy and data-flow mistakes early in CI.

---

## 12. Implementation plan

1. Add AEP-0004 and collect maintainer/community review.
2. Draft SPEC additions for strategy semantics and decision traceability.
3. Add positive/negative fixtures for combination semantics.
4. Add non-normative end-to-end interop fixtures.
5. Update CI gates and conformance docs accordingly.
6. Release v0.5.0 with migration notes as needed.
