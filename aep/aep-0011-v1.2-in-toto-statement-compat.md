# AEP-0011: v1.2 in-toto Statement payloadType Compatibility

- Status: Draft
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v1.2.0 (MINOR)
- Related: AEP-0009 (Accepted), AEP-0010 (Accepted)
- Scope: Schema + fixtures + CI gates (no mandatory verification)

---

## 1. Abstract

This AEP adds optional compatibility for in-toto Statement payloads
carried in DSSE envelopes.

v1.2.0 introduces:

- an optional in-toto Statement profile schema
- DSSE payloadType compatibility:
  - `application/vnd.in-toto+json`
  - `application/vnd.in-toto.<predicate>+json`
- positive and negative fixtures with CI gates
- optional conformance-level extension

Signature verification remains optional.

---

## 2. Motivation

AOP v1.1.0 introduced DSSE envelope packaging with CI-adjudicable
positive and negative gates.

The broader ecosystem commonly uses DSSE where payload is an in-toto
Statement:

- in-toto envelope layer defines in-toto payloadType conventions.
- Sigstore bundle usage commonly carries DSSE with in-toto-compatible
  payload typing.
- SLSA provenance is frequently represented via in-toto Statement
  `predicateType`.

This AEP adds interoperability without breaking v1.0 public API.

---

## 3. Scope and non-goals

### In scope

- `schemas/profiles/in-toto-statement.profile.schema.json`
- DSSE gate extension for in-toto payloadType values
- v1.2 positive and negative fixture families
- optional conformance-level integration

### Non-goals

- mandatory signature verification
- full Sigstore bundle parsing
- forcing in-toto Statement as the only envelope payload model

---

## 4. Proposed schemas

### 4.1 In-toto Statement profile schema

Add:

- `schemas/profiles/in-toto-statement.profile.schema.json`

Baseline shape:

- `_type` MUST be `https://in-toto.io/Statement/v1`
- `subject[]` MUST exist and include at least one item
- `predicateType` MUST be present
- `predicate` MUST be present

### 4.2 DSSE payloadType compatibility

Extend DSSE payloadType acceptance to include:

- `application/vnd.in-toto+json`
- `application/vnd.in-toto.<predicate>+json`

Predicate-specific mapping remains advisory in v1.2.

---

## 5. CI gates

For positive v1.2 DSSE fixtures:

1. DSSE envelope MUST validate against DSSE schema.
2. `payload` MUST base64-decode.
3. Decoded payload MUST parse as JSON.
4. Decoded payload MUST validate as in-toto Statement profile.

For negatives:

- each invalid fixture MUST be rejected by schema or semantic checks.

---

## 6. Conformance integration

Add an optional Level 9 in `CONFORMANCE.md` after fixture and gate
coverage is complete.

Level 9 requires Level 8 plus in-toto Statement DSSE compatibility
checks.

---

## 7. Compatibility

- no mandatory verification policy changes
- additive optional profile and fixture family
