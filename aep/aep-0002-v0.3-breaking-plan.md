# AEP-0002: v0.3.0 Breaking Plan - Governance Core Tightening + Extension Namespace and `unevaluatedProperties`

- Status: Accepted
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.3.0
- Supersedes/Related: AEP-0001 (Accepted), SPEC Governance Definitions (Normative), `schemas/aop-object.schema.json`
- Scope: Specification + normative JSON Schema + conformance fixtures + CI gates

---

## 1. Abstract

This AEP proposes the v0.3.0 change set for Agent Object Protocol (AOP) with two goals:

1. Governance core tightening: when the optional governance sections (`invocation`, `capabilities`, `security`) are present, their core fields become REQUIRED (strong consistency).
2. Extension namespace standardization: define a standard extension namespace (for example, `x-<vendor>-*`) and enable schema-level unknown-field rejection using `unevaluatedProperties: false`, while still allowing extensions through the defined namespace.

These changes are expected to include breaking changes and therefore require explicit migration guidance, per AEP-0001 and SemVer intent.

---

## 2. Normative language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" are to be interpreted as described in RFC 2119.

---

## 3. Motivation

AOP has achieved a schema-backed conformance baseline (v0.2.x) with:

- Normative schema compilation/validation in CI
- Positive and negative fixtures
- Normative governance hooks in SPEC

However, two adoption blockers remain:

- Soft governance: implementers can include `invocation`, `capabilities`, or `security` but omit key fields, reducing interoperability and making policy gating unreliable.
- Unbounded extension surface: without a standard extension namespace, random unknown fields accumulate and cannot be safely rejected.

v0.3.0 addresses these with a focused breaking release.

---

## 4. Definitions

- Governance section present: the manifest contains an object at `invocation`, `capabilities`, or `security`.
- Core fields: minimal fields required to make the section interoperable across runtimes.
- Extension field: a field explicitly permitted by extension namespace rules.
- Unknown field: any field not defined by schema and not permitted as an extension field.

---

## 5. Proposal A - Governance Core Tightening (Breaking)

### 5.1 `invocation` core fields

If `invocation` is present, the following fields become REQUIRED:

- `invocation.mode`
- `invocation.idempotency`

Optional (unchanged): `timeouts_ms`, `retry`, and approved extension fields.

Schema change in `$defs.Invocation`:

- `required: ["mode", "idempotency"]`

### 5.2 `capabilities` core fields

If `capabilities` is present, the following fields become REQUIRED:

- `capabilities.side_effects`
- `capabilities.determinism`

Optional: `token_cost_hint`, plus approved extension fields.

Schema change in `$defs.Capabilities`:

- `required: ["side_effects", "determinism"]`

### 5.3 `security` core fields

If `security` is present, the following becomes REQUIRED:

- `security.auth` (array, `minItems: 1`)

Existing conditional constraints (for example, oauth2 vs none semantics, integrity relations) remain.

Schema change in `$defs.Security`:

- `required: ["auth"]`
- `properties.auth.minItems = 1`

### 5.4 Rationale

These requirements make governance sections mechanically interoperable:

- Runtimes can plan deterministically
- Policies can gate execution reliably
- Conformance suites can assert meaningful behavior

---

## 6. Proposal B - Extension Namespace + `unevaluatedProperties` (Breaking)

### 6.1 Extension namespace rule

AOP defines an extension namespace.

Any object MAY include extension fields matching:

- Top-level: `x-<vendor>-<key>`
- Within governance sections: `x-<vendor>-<key>`

Where:

- `<vendor>` MUST be a lowercase ASCII slug matching `[a-z0-9][a-z0-9-]{0,31}`
- `<key>` MUST match `[a-z0-9][a-z0-9-]{0,63}`

### 6.2 Unknown-field rejection

To prevent unbounded drift, schemas MUST reject unknown fields outside the extension namespace by enabling:

- `unevaluatedProperties: false` at:
  - Manifest top-level
  - `$defs.Invocation`
  - `$defs.Capabilities`
  - `$defs.Security`

### 6.3 Schema mechanics (normative)

- Add `patternProperties` for `^x-[a-z0-9][a-z0-9-]{0,31}-[a-z0-9][a-z0-9-]{0,63}$` with value schema `true` (any JSON type), or a constrained policy if desired.
- Set `unevaluatedProperties: false` to enforce rejection outside known properties and allowed extensions.

---

## 7. Compatibility impact

v0.3.0 is a MINOR release under v0.x and MAY contain breaking changes, but MUST include:

- Breaking change disclosure
- Migration guidance
- Updated examples and fixtures
- Updated schema and CI conformance

Expected breaks:

- Objects with `invocation`, `capabilities`, or `security` present but missing newly required core fields will fail validation.
- Objects containing ad-hoc unknown fields (not in extension namespace) will fail validation once `unevaluatedProperties: false` is enabled.

---

## 8. Migration guide (required for v0.3.0 release)

### 8.1 Governance core tightening migrations

#### A) `invocation` present but missing core fields

Before (v0.2.x valid, v0.3.0 invalid):

```json
{ "invocation": { "timeouts_ms": { "default": 30000 } } }
```

After (v0.3.0 valid):

```json
{
  "invocation": {
    "mode": "request_response",
    "idempotency": "unknown",
    "timeouts_ms": { "default": 30000 }
  }
}
```

#### B) `capabilities` present but missing core fields

Add:

- `side_effects: "unknown"`
- `determinism: "unknown"`

#### C) `security` present but missing auth

Add:

- `auth: [{ "scheme": "none" }]` for public/no-auth objects, or
- `auth: [{ "scheme": "oauth2", ... }]` with required oauth2 relation fields.

### 8.2 Extension namespace migrations

#### A) Unknown top-level field

Before:

```json
{ "foo": "bar" }
```

After (choose one):

- Rename into extension namespace:

```json
{ "x-acme-foo": "bar" }
```

- Or remove if not needed.

#### B) Unknown field inside governance section

Apply the same rule, for example:

- `invocation.foo` -> `invocation["x-acme-foo"]`

### 8.3 Quick migration table

| Area | v0.2.x | v0.3.0 | Minimal migration |
| --- | --- | --- | --- |
| `invocation` present | Core fields may be omitted | `mode` + `idempotency` REQUIRED | Add both fields with valid enum values |
| `capabilities` present | Core fields may be omitted | `side_effects` + `determinism` REQUIRED | Add both fields with valid enum values |
| `security` present | `auth` may be omitted | `auth` REQUIRED (object) | Add `auth` object (`scheme: none` or valid auth scheme payload) |
| Unknown top-level fields | Often tolerated | Rejected unless `x-<vendor>-<key>` | Rename to extension namespace or remove |
| Unknown governance fields | Often tolerated | Rejected unless `x-<vendor>-<key>` | Rename to extension namespace or remove |

## 9. Conformance changes (fixtures + CI gates)

### 9.1 New negative fixtures (expected invalid)

Add invalid fixtures under `examples/invalid/`:

- `aop-invocation-missing-mode.invalid.json`
- `aop-invocation-missing-idempotency.invalid.json`
- `aop-capabilities-missing-side-effects.invalid.json`
- `aop-capabilities-missing-determinism.invalid.json`
- `aop-security-missing-auth.invalid.json`
- `aop-unknown-field-top-level.invalid.json`
- `aop-unknown-field-invocation.invalid.json`

CI MUST reject these fixtures.

### 9.2 New migrated valid fixtures

Add valid fixtures under `examples/` or `examples/migrated/`:

- `aop-invocation-migrated.valid.json`
- `aop-capabilities-migrated.valid.json`
- `aop-extension-x-vendor.valid.json`

CI MUST validate these fixtures as valid.

### 9.3 Validation toolchain

Repository CI uses Ajv CLI with `--spec=draft2020` for schema compilation and validation.

## 10. Alternatives considered

1. Leave governance core optional forever
- Rejected: interoperability and policy gating remain underspecified.

2. Allow arbitrary unknown fields
- Rejected: conformance becomes unenforceable and drift accumulates.

3. Use `additionalProperties: false` only
- Rejected: insufficient under composed schemas; `unevaluatedProperties` is the correct post-evaluation mechanism in 2020-12.

---

## 11. Security considerations

- Requiring governance core fields reduces ambiguity that can lead to confused-deputy behavior.
- Enforcing extension namespaces reduces the risk of hidden shadow fields that bypass policy checks.
- Unknown-field rejection improves auditability and deterministic parsing.

---

## 12. Implementation plan (repository steps)

1. Update `schemas/aop-object.schema.json`:
- Add required fields for governance cores when sections are present
- Add extension `patternProperties`
- Set `unevaluatedProperties: false` at top-level and in governance defs

2. Update `SPEC.md`:
- Mark core fields REQUIRED when section present
- Document extension namespace rules

3. Add fixtures (invalid + migrated valid).

4. Ensure CI gates pass (compile, validate valid, reject invalid).

5. Tag and release `v0.3.0` with explicit migration notes.
