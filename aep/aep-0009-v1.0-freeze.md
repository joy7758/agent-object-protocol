## 1. Abstract

This AEP defines the v1.0.0 freeze for AOP:

- v1.0.0 defines the public API and makes stability guarantees
  under SemVer.
- The public API is the set of schemas, profiles, and conformance
  levels listed in `V1_PUBLIC_API_CANDIDATE.md`.
- Changes to the public API after v1.0.0 MUST use the AEP process
  and follow SemVer versioning rules.

SemVer states that `1.0.0` defines the public API, while `0.y.z`
versions are not considered stable.

---

## 2. Motivation

AOP now has:

- Mature core schema families (object/policy/decision/evidence + profiles)
- CI-adjudicable conformance (positive fixtures pass, negative fixtures reject)
- Evidence binding based on `SHA-256(JCS(json))`

The goal of v1.0.0 is to give adopters a stable contract that can be
implemented independently and validated deterministically.

RFC 8785 defines JCS as a hashable canonical representation suitable for
cryptographic methods, which underpins AOP digest binding.

---

## 3. Definitions

- Public API (AOP): the normative schemas, profiles, and conformance
  requirements explicitly listed in `V1_PUBLIC_API_CANDIDATE.md`.
- Non-normative artifacts: interoperability guidance and schemas that are
  not promoted to normative status by an Accepted AEP.
- Conformance suite: repository schemas, examples, invalid fixtures, and CI
  gates that adjudicate correctness.

---

## 4. v1.0 Freeze Scope

### 4.1 Frozen (v1 public API)

Upon v1.0.0, the following are frozen public API:

- `schemas/aop-object.schema.json`
- `schemas/aop-policy.schema.json`
- `schemas/aop-policy-decision.schema.json`
- `schemas/aop-evidence.schema.json`
- `schemas/profiles/*` that are listed in `V1_PUBLIC_API_CANDIDATE.md`
- Conformance Levels 2-7 and their gate semantics in `CONFORMANCE.md`
- Evidence digest rule: `SHA-256(JCS(json))` as defined by RFC 8785

### 4.2 Not frozen (explicitly excluded from v1 public API)

- Registry schemas remain non-normative unless promoted by a future AEP.
- `x-<vendor>-<key>` fields remain extension points.
- Signature verification remains optional at v1.0.
  DSSE/in-toto envelope alignment MAY be adopted later.

---

## 5. Compatibility Contract (SemVer for AOP)

After v1.0.0:

- MAJOR: breaking changes to frozen public API or conformance semantics.
- MINOR: backward-compatible additions (optional fields, new optional
  profiles, additive conformance levels).
- PATCH: backward-compatible fixes that do not alter public API surface.

This follows the SemVer contract for public API compatibility.

---

## 6. Governance and Change Control

### 6.1 Required process

Any change that modifies a v1 public API artifact MUST:

- be proposed via AEP (Draft -> Accepted),
- include migration notes when applicable,
- include conformance updates (fixtures + gates).

### 6.2 Freeze discipline

From v1.0.0 release-candidate stage:

- No new public API fields are added without an Accepted AEP targeting v1.
- CI signal quality remains strict (no global lint relaxation).

---

## 7. Evidence and Provenance Policy (v1)

AOP v1.0.0 treats evidence as first-class:

- Provenance is verifiable information about artifacts
  (where/when/how produced).
- Evidence binding MUST remain CI-adjudicable via JCS + SHA-256.
- Profile validation MUST remain enforced for profiled evidence scenarios.

---

## 8. Release Plan

### 8.1 Work items for v1.0.0

1. Finalize `V1_PUBLIC_API_CANDIDATE.md` and reference it from README.
2. Ensure all frozen schema families have compile-pass, valid-pass, and
   invalid-reject gates.
3. Keep Conformance Levels 2-7 stable and documented.
4. Prepare `RELEASE_NOTES_v1.0.0.md` including:
   - public API definition
   - stability guarantees
   - explicitly non-normative items

### 8.2 Tag and release

Create an annotated `v1.0.0` tag and publish a GitHub Release from
`RELEASE_NOTES_v1.0.0.md`.
