# AEP-0005: v0.6.0 Agenda - Suite Expansion (Combo + MCP)

- Status: Accepted
- Authors: joy7758 (maintainers may append)
- Created: 2026-03-04
- Target release: v0.6.0
- Related: AEP-0004 (Accepted), CONFORMANCE.md, V0_5_SEMANTICS.md

---

## 1. Abstract

v0.6.0 expands v0.5 semantics from single-case demos into a reusable
conformance suite:

- Combo suite expansion: add multiple policy/object strategy cases.
- MCP suite expansion: add multiple tool/output schema families.
- Keep the same discipline: schema compile + expected pass + invalid
  reject + CI gate.

No runtime implementation is required in this repository.

---

## 2. Goals

1. Prove coverage, not just demonstration.
2. Keep fixtures machine-verifiable and regression-friendly.
3. Preserve v0.5 conformance workflow and naming discipline.

---

## 3. Scope and non-goals

### In scope (v0.6.0)

- Two new combo cases with strategy-delta expected decisions.
- Two new MCP tool cases with schema-derived output validation.
- Negative fixtures for each new case.
- CI expansion to include v0.6 fixture paths.

### Out of scope (v0.6.0)

- Runtime execution engine implementation.
- Registry protocol hard guarantees (hold for later phases).
- Mandatory single strategy across all runtimes.

---

## 4. Four concrete cases (execution checklist)

### Case C1 (combo): first-match vs deny-overrides

Intent:

- Demonstrate explicit decision delta between `first_match` and
  `deny_overrides` for the same policy/object pair.

Files:

- `examples/v0.6/combo/object/c1-sensitive-writer.object.json`
- `examples/v0.6/combo/policy/c1-permit-vs-deny.policy.json`
- `examples/v0.6/combo/expected/c1-permit-vs-deny.first_match.expected.json`
- `examples/v0.6/combo/expected/c1-permit-vs-deny.deny_overrides.expected.json`
- `examples/v0.6/combo/expected/c1-permit-vs-deny.first_match.semantic.invalid.json`

Required checks:

- Both expected files are schema-valid and semantic-valid.
- Semantic-invalid file is schema-valid but semantic-rejected by combo
  gate.

### Case C2 (combo): permit-overrides vs deny-unless-permit

Intent:

- Demonstrate explicit decision delta between `permit_overrides` and
  `deny_unless_permit` for the same policy/object pair.

Files:

- `examples/v0.6/combo/object/c2-network-sensitive.object.json`
- `examples/v0.6/combo/policy/c2-permit-vs-deny.policy.json`
- `examples/v0.6/combo/expected/c2-permit-vs-deny.permit_overrides.expected.json`
- `examples/v0.6/combo/expected/c2-permit-vs-deny.deny_unless_permit.expected.json`
- `examples/v0.6/combo/expected/c2-permit-vs-deny.permit_overrides.semantic.invalid.json`

Required checks:

- `permit_overrides` expected and `deny_unless_permit` expected both
  pass schema and semantic checks.
- Semantic-invalid fixture is rejected by combo semantic gate.

### Case M1 (mcp): kv-get

Intent:

- Add a compact key-value retrieval output with scalar unions to improve
  output-shape coverage beyond list-oriented examples.

Files:

- `examples/v0.5/mcp/tool/kv-get.tool.object.json`
- `examples/v0.5/mcp/expected/kv-get.output.expected.json`
- `examples/invalid/v0.5/mcp/kv-get.output-missing-value.invalid.json`
- `examples/invalid/v0.5/mcp/kv-get.output-wrong-type.invalid.json`

Required checks:

- Expected output validates against derived `tool.schema.outputs`.
- Invalid outputs are rejected by MCP gate.

### Case M2 (mcp): web-fetch

Intent:

- Add a tool with nested output structure and response-status
  constraints to increase schema-derived validation depth.

Files:

- `examples/v0.5/mcp/tool/web-fetch.tool.object.json`
- `examples/v0.5/mcp/expected/web-fetch.output.expected.json`
- `examples/invalid/v0.5/mcp/web-fetch.output-status-out-of-range.invalid.json`
- `examples/invalid/v0.5/mcp/web-fetch.output-missing-body.invalid.json`

Required checks:

- Expected output validates against derived `tool.schema.outputs`.
- Invalid outputs are rejected by MCP gate.

## 5. CI extension points

### Combo gate

- Keep v0.5 combo gate unchanged and add a dedicated v0.6 combo gate
  for `examples/v0.6/combo/...`.
- Keep semantic checker pattern-based by case prefix:
  - `c1-permit-vs-deny.*`
  - `c2-permit-vs-deny.*`

### MCP gate

- Extend MCP tool discovery to include:
  - `examples/v0.5/mcp/tool/*.json`
  - additional tool prefixes under `examples/v0.5/mcp/tool/*.json`
- Keep existing per-tool prefix derivation for expected/invalid
  matching.

## 7. Acceptance criteria (v0.6.0)

v0.6.0 suite-expansion is complete when:

1. Case C1/C2 fixtures are present and validated by schema + semantic
   combo gate.
2. Case M1/M2 fixtures are present and validated by MCP derived-schema
   gate.
3. New invalid fixtures are rejected in CI.
4. Conformance documentation references v0.6 suite paths.

Current status: **All v0.6.0 acceptance criteria satisfied**.
